# Why was fastai + MT-heads behind mini_trainer's own loop?

**Status:** RESOLVED (2026-07-16). Gap closed and overtaken: **0.8976 vs 0.896**.
**Metric:** species (level 0) macro-F1 on the global **test** fold (`set == '0'`), via
mini_metrics. Every number below is that metric unless stated.

## The question

Running mini_trainer's hierarchical heads inside fastai's training loop (dev/030) scored
**0.8297**, while mini_trainer's own loop reached **0.896** on the same data and the same
architecture. Same heads, same backbone, same dataset — so the ~6.6 point gap was somewhere in
the *training loop*, and the goal was to close it without touching model architecture, image
resolution, batch size, or epoch budget.

## The answer: it was under-annealing, mostly

The ladder, generated by `python dev/036_ledger.py --chain --only scored`:

| run | change | test F1 |
|---|---|---|
| `20260712-072542` | Muon + flat_cos, heavy aug, grad_clip 1.0 | 0.8297 |
| `20260713-164456` | + warmup 0.5ep, light aug, grad_clip 5.0 *(bundled)* | 0.8769 |
| `20260714-072404` | flat_cos → **one_cycle** | 0.8887 |
| `20260716-105029` | same recipe, **10 epochs** | **0.8976** |

**The single biggest lever was the LR schedule shape.** `flat_cos` holds the LR flat and only
spends its final 25% on the cosine anneal — the model was still converging hard when the run
ended, so it was being graded mid-descent. `one_cycle` anneals over ~90% of the run and bought
~1.2 points on its own, changing nothing else.

**The second lever was simply the epoch budget.** mini_trainer's real budget turned out to be
**10 epochs, not 5** — the comparison had been unfair from the start. Matching it bought
another ~0.9 points. Worth stating plainly: a chunk of the "gap" was never a gap, it was a
misread baseline. Check the baseline's actual settings before optimising against its number.

At 10 epochs there was still no overfitting — valid_loss fell 0.4472 → 0.4327 and val F1 rose
0.8950 → 0.8977 across the last two epochs. **The 10-epoch run was still improving when it
stopped**, so more epochs would likely still pay. That is the cheapest untested lever left,
and it is only "untested" because the budget was fixed by the comparison, not by evidence.

## Negative results (do not pay for these twice)

- **Class-distribution regularisation is a wash.** `class_reg_strength=0.001`
  (`20260714-194321`) scored val 0.8860 against the equivalent unregularised run's 0.8880 —
  inside noise. No effect, don't retry without a reason.
- **The 0.92-val-vs-0.83-test "fold bug" did not exist.** Folds 0 and 1 are statistically
  identical (~633k images each, 12041 species, instance accuracies match at every level). It
  was a metric-definition mix-up: dev/030's training-log `F1(macro)` averages **all three
  taxonomic levels** (inflated by easy genus/family), while mini_metrics' "level 0 f1" is
  **species only**. Like-for-like, both were ~0.83. Also `acc_speciesKey` (fastai) ==
  `micro_accuracy` (mini_metrics), *not* its `accuracy` column, which is macro.
  **Lesson: before hunting a bug, check that both numbers mean the same thing.**

## Landmines found along the way

- **Muon (`MuonAuxAdamW`) re-partitions param groups**, which breaks fastai's freeze
  bookkeeping — it only works with unfrozen schedules. It also takes tuple betas, so fastai's
  stock `fit_one_cycle` chokes on it (that schedules momentum as well as LR). Hence the
  hand-built **LR-only** schedules in dev/030 (`warmup_cos_schedule`, `front_loaded_schedule`).
- **`SaveModelCallback` defaults to `with_opt=False`** — checkpoints carry weights only, no
  optimizer state. A resume restarts Muon's momentum cold, which costs a few batches of
  re-warming, not correctness.
- **`gen_df`'s cache path ignores `fold`, `min_img_per_spc`, and `family_filter`** (dev/028).
  Change the filtering and you silently reuse the wrong cache. Hasn't bitten yet — every
  recent run used identical filtering — but it is a live correctness landmine.

## Final numbers (`20260716-105029`, test fold, 629,742 images, 12,041 species)

| level | macro-F1 | micro-acc |
|---|---|---|
| species | **89.76%** | 94.09% |
| genus | 95.69% | 97.42% |
| family | 97.36% | 99.38% |

Winning recipe: efficientnet_v2_s, independent heads, Muon, `one_cycle`, `warmup_epochs: 0.5`,
`grad_clip: 5.0`, light aug (no warp, no lighting), 10 epochs, bs 64, 460→256.
Config: `configs/20260715_heads_global_independent_muon_onecycle_10ep.yaml`.

This run also survived a mid-training machine hang (see
[2026-07-gpu-hang.md](2026-07-gpu-hang.md)) via dev/030's `resume_checkpoint` /
`resume_epochs_done` + `fit_resume`, which rebuild the original LR curve and continue from the
exact fractional position rather than restarting the anneal.

## Loose ends

- **Never ablated** the three changes bundled in `20260713-164456` (warmup / light aug /
  grad_clip 5.0). They moved together and bought 4.7 points as a group; nobody knows the split.
- **10 epochs was still improving** — the budget was never the binding constraint, the
  comparison was.
- **Long-tail interventions** are the open thread → [2026-07-does-longtail-help.md](2026-07-does-longtail-help.md).
